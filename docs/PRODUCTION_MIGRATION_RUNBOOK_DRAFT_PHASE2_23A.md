# Phase 2.23A 生产迁移 Runbook 草案

本文件是上线前迁移流程草案，仅用于人工审查。本阶段未连接生产库，未执行任何迁移。

## 通用前置要求

1. 明确目标数据库环境：预发布 / 生产。
2. 确认使用正式环境变量，不使用本地 `.local/env`。
3. 备份目标数据库。
4. 锁定当前代码版本和 migration 文件。
5. 人工审查 migration SQL。
6. 准备回滚方案。
7. 准备上线后验证清单。

## A. 生产库为空库

适用条件：

- 生产库是全新空库。
- 没有历史表、历史数据、旧订单、旧会员、旧库存。

建议步骤：

1. 备份确认
   - 即使为空库，也记录当前状态和连接信息。
2. 审查 initial migration
   - 以 `prisma/migrations/20260614_phase2_23a_initial_release_candidate_schema/migration.sql` 作为参考。
   - 人工确认无危险 SQL。
3. 在预发布空库演练
   - 执行 reviewed initial migration。
   - 验证 27 个模型对应表、14 个 enum、索引和外键。
4. 执行正式 seed
   - 只运行正式基础 seed。
   - 不运行本地演示管理员、演示库存、legacy demo 清理等本地逻辑。
5. 验证基础数据
   - 门店、商品、SKU、库存、优惠券模板、会员价。
6. 验证核心链路
   - 商品列表、SKU、购物车、checkout、优惠券、下单、订单详情。
7. 支付专项验证
   - 只在支付准生产或正式验收窗口内验证。
   - 确认回调验签、订单状态、优惠券 USED、幂等。
8. 记录上线结果
   - 记录 SQL 版本、执行时间、执行人、验证结果。

注意：即使为空库，也不建议直接用 `prisma db push` 上生产。

## B. 生产库已有旧表或旧数据

适用条件：

- 生产库已经有历史表或历史数据。
- 曾经跑过旧版本 schema。
- 有历史订单、商品、库存、顾客、优惠券等数据。

这种情况不能直接执行 initial migration。

建议步骤：

1. 完整备份
   - 备份 schema 和数据。
   - 验证备份可恢复。
2. 导出现有数据库 schema
   - 使用只读方式获取当前表、字段、索引、约束、enum。
3. 生成差异 migration
   - 从 current-db 到 `prisma/schema.prisma` 生成 diff。
   - 不要直接执行。
4. 人工审查差异 SQL
   - 检查是否有 `DROP`、`TRUNCATE`、`DELETE`。
   - 检查新增 NOT NULL 字段。
   - 检查唯一约束和历史重复数据。
   - 检查外键和孤儿数据。
5. 数据修复计划
   - 如有重复数据，先准备去重策略。
   - 如有缺失字段，准备回填脚本。
   - 如有 enum 状态不一致，准备状态映射。
6. 预发布库演练
   - 使用生产备份恢复到预发布库。
   - 在预发布库执行迁移和数据修复。
   - 跑完整顾客端回归。
7. 回滚方案
   - 明确数据库恢复方式。
   - 明确代码回滚版本。
   - 明确支付回调和订单状态中断处理。
8. 生产执行
   - 在维护窗口执行。
   - 执行后立即验证关键链路。

## 上线后验证清单

- Prisma schema 与数据库结构一致。
- 门店、商品、SKU、库存可读。
- 商品列表、SKU 弹窗、购物车、checkout 正常。
- 会员价和优惠券 quote-preview 正常。
- 创建订单后金额以后端为准。
- 待支付订单锁券。
- 取消订单释放券。
- 支付成功后券变 USED。
- 支付回调幂等。
- 后台管理员登录可用。
- 后台订单、库存、商品、门店页面可读。

## 本阶段结论

Phase 2.23A 仅生成迁移草案和迁移 runbook，不执行任何生产动作。

下一步建议进入 Phase 2.23B 或 Phase 2.24A 前，先确认目标上线数据库是空库还是已有库。
