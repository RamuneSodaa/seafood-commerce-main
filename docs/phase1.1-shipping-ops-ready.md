# Phase 1.1A Shipping Ops Ready

本文档用于说明当前仓库在 Phase 1 基础上，围绕“最小真实发货运营闭环”已经完成的能力、仍未完成的边界，以及后续进入 Phase 1.1B 前的承接点。

## 1. 本阶段目标

Phase 1.1A 的目标不是扩展新业务，而是在现有 `SHIPPING` 订单链路上，把“能演示”推进到“最小可真实操作”：

- 顾客侧仍按现有方式创建并支付邮寄订单
- 后台能够识别待发货订单
- 后台能够录入真实物流信息，而不是继续使用硬编码默认值
- 发货后订单状态、物流信息和送达动作可以继续沿现有链路推进

本阶段保持以下边界不变：

- 不改后端接口契约
- 不新增地址簿
- 不接微信支付
- 不改路由结构
- 不扩到新的业务模块

## 2. 已完成内容

围绕发货运营闭环，当前已经完成：

- storefront 已具备邮寄订单下单与订单详情展示能力
- API 已具备 `mark-paid`、`ship`、`deliver` 等现有发货链路接口
- admin 订单详情页已从“硬编码发货按钮”升级为“最小发货表单”
- 发货表单已支持人工录入：
- `快递公司`
- `运单号`
- 发货提交时要求两个字段都不能为空
- 发货提交中按钮会禁用，避免重复点击
- 发货成功后会刷新当前订单详情，并立即显示最新物流信息
- admin 订单作业台已补充前端筛选视图
- 当前支持的筛选项：
- `全部`
- `待发货`
- `已发货`
- `已送达`
- `自提单`
- `待取货`
- 筛选结果为空时，后台会显示中文空状态提示

## 3. 当前业务闭环

当前可验证的邮寄发货闭环如下：

1. 顾客在 storefront 选择商品并提交 `SHIPPING` 订单。
2. 顾客完成支付标记后，订单进入 `PAID_PENDING_SHIPMENT`。
3. 后台订单作业台可通过“待发货”筛选快速识别该订单。
4. 运营或门店人员进入订单详情页，在发货录入区填写：
   - 快递公司
   - 运单号
5. 点击“确认发货”后，前端继续调用现有 `ship` 接口。
6. 页面刷新后，订单进入 `SHIPPED`，物流公司、运单号、发货时间可立即查看。
7. 后台继续使用现有“标记已送达”动作，将订单推进到 `DELIVERED`。

对应主路径可以概括为：

`顾客下单 -> 支付 -> 后台待发货 -> 录入物流 -> 发货 -> 送达`

## 4. 前后台分别已具备的能力

### storefront 已具备

- 商品列表、商品详情、下单页、订单列表、订单详情的完整演示路径
- 顾客可创建 `SHIPPING` 订单
- 顾客可查看订单状态、履约方式、收货地址和物流信息
- 订单详情页可展示 `已发货 / 已送达` 等中文状态与时间线

### admin 已具备

- 订单作业台可拉取现有订单列表
- 可按“待发货 / 已发货 / 已送达 / 自提单 / 待取货”做前端筛选
- 订单详情页可查看状态、门店、商品、金额、收货地址、物流跟踪和状态日志
- 对 `PAID_PENDING_SHIPMENT` 的邮寄订单，可人工录入快递公司和运单号后确认发货
- 对 `SHIPPED` 订单，可继续使用现有动作标记已送达

## 5. 当前仍未完成内容

本阶段刻意没有做，且仍留给后续阶段处理的内容：

- 地址体系收口
- 顾客地址簿
- 多地址管理
- 地址选择与默认地址策略
- 后台更完整的物流作业能力
- 搜索
- 高级筛选组合
- 批量发货
- 打印面单
- 微信支付或真实支付回调
- 更复杂的物流轨迹同步

这些内容不属于当前 Phase 1.1A 的阻塞项。

## 6. 手动验收步骤

### 本地启动

1. 启动 API：

```bash
npm run start -w @seafood/api
```

2. 启动 storefront：

```bash
npm run dev -w @seafood/storefront-web
```

3. 启动 admin：

```bash
npm run dev -w @seafood/admin-web
```

### 验收一笔 SHIPPING 订单

1. 在 storefront 创建一笔 `SHIPPING` 订单。
2. 调用现有支付动作或脚本，让订单进入 `PAID_PENDING_SHIPMENT`。
3. 打开 admin `订单作业台` 页面，切换到“待发货”筛选。
4. 进入该订单详情页。
5. 在“发货录入”里填写：
   - 快递公司，例如 `顺丰速运`
   - 运单号，例如 `SF1234567890`
6. 点击“确认发货”。
7. 确认页面自动刷新，且立即看到：
   - 订单状态变为 `已发货`
   - 物流公司已显示
   - 运单号已显示
   - 发货时间已显示
8. 点击“标记已送达”。
9. 确认订单状态更新为 `已送达`。
10. 回到 storefront 订单详情，确认顾客侧也能看到对应物流状态变化。

### 可用脚本

- `./phase1-shipping.sh <order-id> <customer-id> [paid-amount-cents]`

该脚本可用于辅助验证邮寄订单从支付到发货链路的主要步骤。

## 7. 下一阶段建议

建议把下一阶段命名为 `Phase 1.1B Address Ready`，但范围只推进到“地址体系收口”，不要扩展到支付体系。

下一阶段更合理的最小目标：

- 梳理并固定 `SHIPPING` 订单所需的地址字段口径
- 明确 storefront 下单页的地址录入边界
- 为后续地址簿或地址管理做准备

当前不建议在 Phase 1.1B 里同时展开：

- 微信支付
- 小程序接入
- 新履约模块
- 复杂物流平台对接

## 8. 阶段结论

当前仓库已经可以视为：

- `Phase 1` 主链路完成
- `Phase 1.1A` 最小真实发货运营闭环已就绪

如果需要一个更准确的阶段名，建议使用：

- 首选：`phase1.1a-shipping-ops-ready`
- 备选：`phase1.1a-min-shipping-closed`

建议优先使用 `phase1.1a-shipping-ops-ready`，它最准确表达当前状态：发货运营链路已经具备最小真实录入和后台处理能力，但尚未进入地址体系的下一阶段收口。
